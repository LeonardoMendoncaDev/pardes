This is a simple rule to apply before every `git add`:

> **If you cannot explain in three sentences why the code does what it does, you did not deserve to accept the output. Review.**

The rule seems obvious until the moment one tries to apply it for a full week. That is when one realises that a considerable part of the code we accept from the AI agent does not survive this minimum criterion — and never did. We did not notice because we were never forced to articulate.

## Why this matters

The agent produces syntactically correct code almost always. It compiles, passes the *linter*, looks reasonable. But it may be semantically wrong — wrong algorithm, violated invariant, ignored *edge case*, assumption about *input* that does not hold. The *linter* catches none of that.

The human is the last layer of semantic validation before the code enters the repository. The agent has no responsibility for the code; the programmer does. And *git blame* points to the programmer, not to the model that generated the text.

From this framing, the three-sentence test starts to make sense as a practical discipline.

## Why three

It is a specific choice, not arbitrary:

- **One sentence** lets you mask lack of understanding. *"This function processes the input"* explains nothing.
- **Ten sentences** let you hide. If you need ten sentences to explain a function, the function is confused or you did not understand it.
- **Three sentences** are enough to articulate what the code does, why it is structured this way, and what would happen if an important input changed.

It is the shortest functional threshold where real understanding can still be demonstrated.

## The three questions the sentences must cover

### 1. What the code does

First sentence. No jargon, no metaphors. In domain language if the code is business logic; in technical language if it is infrastructure.

It is not enough to say *"handles the request"*. It is enough to say *"validates the CPF, queries the balance in cache, accepts if ≥ 100, rejects otherwise, logs the event"*.

### 2. Why it is structured this way

Second sentence. This is the one that most frequently catches the programmer off guard. It is not *"because it makes sense"* — it is why it is structured **this** way, and not some other.

*"I use `Promise.all` instead of sequential `await` because the three fetches do not depend on each other, so the total time is the slowest one, not the sum."* That is a defensible second sentence.

### 3. What would happen if an important input changed

Third sentence. Forces the brain to simulate the future. What happens if the input starts arriving as `null`? If the downstream service becomes slower? If someone starts calling this method 100 times per second instead of once?

If the answer is *"I don't know"*, the code will not survive the first real change in the environment.

<div class="exercise">

**To apply now**

Pick the last *commit* where an AI agent helped you write code. Open the *diff*.

Try to write three sentences covering: what it does, why it is structured this way, what happens if an important input changes.

If you need more than three sentences, do not expand the text — simplify the code.

<details>
<summary>How to tell if the answer is good enough?</summary>

Read the three sentences and ask yourself whether you would defend them in a code review with a colleague who challenges each of them. If you hesitated, it is not because the review is scary — it is because understanding is not yet firm.

</details>
</div>

## Common pitfalls

<div class="trap">

**Feeling that you know without being able to articulate**

The human brain has a well-documented capacity to feel it understands without actually understanding. Articulation out loud — or in writing — is one of the most reliable tests to distinguish real understanding from the illusion of understanding.

</div>

<div class="trap">

**Three long sentences instead of three normal sentences**

Clauses chained with *"which also"* and *"furthermore"* turn three sentences into ten in disguise. If the code requires long compound sentences to be explained, the problem is almost always with the code.

</div>

<div class="trap">

**Applying the rule only to "important" code**

Introduces an invisible hierarchy: important code versus secondary code. In practice, the problem that wakes the programmer up at three in the morning is almost always in code that nobody found important enough to review carefully.

</div>

## When the test is failed

It happens — fairly often at the start. What to do:

- **Do not delete the code.** It is expensive and nine out of ten times unnecessary.
- **Ask the agent to explain line by line.** Ask about each block of three or four lines what it does and why it is there.
- **Rewrite parts until the mental model settles.** Often the rewriting exercise clarifies.
- **Simplify the code until it fits in three sentences.** This is the option that most frequently produces the best result.

## On the tone of the rule

The formulation *"you did not deserve to accept the output"* is deliberately firm. The objective is to install a small friction in the routine — enough to break the habit of accepting by inertia. In day-to-day practice, it applies with less theatricality than the text suggests: one simply pauses, writes the three sentences mentally, and moves on.

<div class="checklist">

- I read the code line by line before *committing*
- I can explain **what** it does in one short sentence
- I can explain the **why** of the structure in one short sentence
- I can predict the **behaviour** under change in one short sentence
- If a colleague asks me tomorrow, I defend the code without hesitation

</div>
